# 手机号授权「新组件」改动清单（先不改代码，仅列需改内容）

## 一、你说的「新组件」指什么？

微信小程序目前有两种手机号授权能力：

| 项目 | 当前用的 | 新组件（可选） |
|------|----------|----------------|
| 名称 | **手机号快速验证组件** | **手机号实时验证组件** |
| open-type | `getPhoneNumber` | `getRealtimePhoneNumber` |
| 事件名 | `bindgetphonenumber` / `@getphonenumber` | `bindgetrealtimephonenumber` / `@getrealtimephonenumber` |
| 回调内容 | `code` + 可能带 `encryptedData`、`iv` | **仅 `code`**，不返回 encryptedData/iv |
| 后端换手机号 | 用 code 调 `phonenumber.getPhoneNumber`，或旧方式用 encryptedData+iv+session_key 解密 | **只能用 code** 调 `phonenumber.getPhoneNumber` |
| 验证方式 | 不保证实时验证 | 每次请求都会实时验证（可能发短信） |
| 计费 | 0.03 元/次（SPU 10000077） | 0.04 元/次（SPU 10000086） |

下面按「若改用**手机号实时验证组件**」列出需要动到的所有地方（先不改代码，只做清单）。

---

## 二、需要改动的文件与内容

### 1. 小程序端：`pages/index/index.vue`（主登录页）

**模板：**

- 把 `open-type="getPhoneNumber"` 改为 `open-type="getRealtimePhoneNumber"`。
- 把 `@getphonenumber="onGetPhoneNumber"` 改为 `@getrealtimephonenumber="onGetPhoneNumber"`（或新函数名如 `onGetRealtimePhoneNumber`，再在内部复用同一套登录逻辑）。

**脚本：**

- **成功判断**：当前是 `e.detail.errMsg !== 'getPhoneNumber:ok'`。若改用实时验证，成功时为 `getRealtimePhoneNumber:ok`，需改为兼容两种或只判断一种，例如：
  - 若只支持新组件：`e.detail.errMsg !== 'getRealtimePhoneNumber:ok'`；
  - 若同时兼容两种：  
    `e.detail.errMsg !== 'getPhoneNumber:ok' && e.detail.errMsg !== 'getRealtimePhoneNumber:ok'`。
- **取参**：当前从 `e.detail` 取 `code`、`encryptedData`、`iv`。实时验证**只返回 `code`**，不再有 `encryptedData`、`iv`。需：
  - 只取 `phoneCode = e.detail.code`；
  - 传给后端的 payload 里**不再传** `encryptedData`、`iv`（或传空），只传 `code`（wx.login）+ `phone_code`（手机号组件返回的 code）。
- **登录成功后的兜底逻辑**：当前有一段「若后端没返回 11 位手机号且前端有 encryptedData/iv，则调 `decryptPhone` 解密并更新手机号」。  
  改用实时验证后，回调里没有 encryptedData/iv，这段逻辑**永远不会成立**，可以：
  - 保留代码不动（相当于死分支），或
  - 删掉/注释掉「用 encryptedData、iv、decryptPhone」这一段，避免误导。

**小结：**  
index.vue 要改的是：组件类型与事件名、成功 errMsg 判断、只使用 `code` 传后端、不再依赖 encryptedData/iv 及 decryptPhone 兜底。

---

### 2. 接口层：`api/auth.js`

- **wechatLogin** 当前入参包含：`code`、`phone_code`、`encryptedData`、`iv`、`nickName` 等。
- 若后端**只用 `phone_code` 调微信「获取手机号」接口**换手机号，则改用实时验证后：
  - 前端仍传 `code`、`phone_code`、`nickName` 即可；
  - `encryptedData`、`iv` 可不再传或传空，后端忽略即可。
- **需要确认**：你们后端登录接口是否**必须**要 encryptedData/iv，还是已经支持「仅 phone_code 换手机号」。若已支持仅 code，auth.js 只需保证调用 wechatLogin 时不再依赖 encryptedData/iv（或传空）；若后端仍依赖解密，则需**后端先改为用 phone_code 换号**再切新组件。

---

### 3. 接口层：`api/user.js`

- **decryptPhone**：当前用于「getPhoneNumber 返回的 encryptedData+iv + wx.login 的 code 解密手机号」。
- 改用**实时验证**后，前端不再拿到 encryptedData/iv，登录流程里**不会再调用** decryptPhone 做手机号获取。
- 若其它页面（如设置页改绑手机号）仍用旧版 getPhoneNumber + 解密，可保留 decryptPhone；若全项目都改为实时验证且不再解密，可考虑后续再删或标记废弃。
- **结论**：为切到新组件，user.js 可以**先不改**，只保证登录流程不再走 decryptPhone 即可。

---

### 4. 后端（需你方确认/修改）

- **换手机号方式**：必须改为**仅用 `phone_code` 调微信服务端接口**  
  `POST https://api.weixin.qq.com/wxa/business/getuserphonenumber?access_token=ACCESS_TOKEN`  
  Body: `{ "code": "前端传的 phone_code" }`  
  获取手机号。  
  实时验证的 code **不能**用 encryptedData+iv+session_key 解密，只能通过该接口换号。
- **登录接口入参**：若当前接收 `phone_code`、`encryptedData`、`iv`，可改为：
  - 优先用 `phone_code` 换手机号；
  - 不再依赖 `encryptedData`、`iv`（收到也忽略）。
- **文档/契约**：若有接口文档或前端类型定义写了「encryptedData/iv 必填」，需改为「仅 phone_code 必填（手机号登录时）」。

---

### 5. 其它使用 wechatLogin / 手机号的地方

- **subPackages/page1/pages/index/index.vue**：当前是 `@tap="handleWeChatLogin"`，走的是 wx.login 的 code，**没有**用 getPhoneNumber，因此若只把「主登录页」改成实时验证，这里**不用改**。
- **subPackages/page2/pages/debug/**：若调试页有直接调 wechatLogin( code ) 或 wechatLogin( { code, phone_code } )，需确认传参是否还带 encryptedData/iv；若只做登录调试，一般只传 code（或 code+phone_code）即可，不涉及解密则可不改。
- 全项目搜索 `getPhoneNumber`、`getphonenumber`、`phone_code`、`decryptPhone` 后，只有**主登录页**和上面几处与「一键手机号登录」强相关，其余为地址/积分等业务用手机号，不依赖本次组件切换。

---

### 6. 公众平台与计费

- 在 **微信公众平台 → 付费管理** 中：
  - 当前若只买了「手机号快速验证组件」（10000077），改用实时验证后需购买 **「手机号实时验证组件」**（10000086），否则会报额度不足（如 errno 1400001）。
  - 两种能力**独立计费、独立额度**，不能混用。

---

### 7. 基础库与兼容

- 实时验证组件有**最低基础库版本**要求，需查看[官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/getRealtimePhoneNumber.html)中的「最低版本」。
- 若需要同时兼容「快速验证」和「实时验证」（例如按基础库或配置切换），则需在 index.vue 里同时处理两种事件名和两种 errMsg，并区分是否带 encryptedData/iv。

## 四、注意事项（与「验证码」的关系）

- **实时验证** = 平台**每次**都会对用户选择的手机号做实时验证（可能发短信），所以若你希望「少发验证码」，**不建议**从快速验证改成实时验证；实时验证反而更容易触发短信验证。
- 若你的诉求是「用新组件、但不想用户收验证码」，需要看微信是否提供了其它**不强制实时验证**的新能力；当前官方文档里，和「手机号授权」相关的新组件主要就是「手机号实时验证」，和「快速验证」是二选一，不能混用同一次回调。

以上为「若改用手机号实时验证组件」时，需要改动的文件与内容清单；按你的要求**先不改代码**，只做对照和评估用。若你指的「新组件」是别的能力（例如其它开放文档里的新接口），可以说下名称或链接，再按那个能力列一版清单。
