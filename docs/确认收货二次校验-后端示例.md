# 微信确认收货组件 · 后端二次校验示例（2026 流程）

## 一、流程说明

1. 前端在 `App.onShow` 收到 `referrerInfo.appId === 'wx1183b055aeec94d1'` 且 `status === 'success'` 后，调用后端 `POST /order/confirm-receive`，传 `order_number`、`transaction_id`（可选但推荐）。
2. **后端必须二次校验**：先调微信「查询订单发货状态」接口，确认微信侧为「已收货/交易完成」后，再更新业务订单状态。
3. **幂等**：用 `order_number` 或 `transaction_id` 做幂等，同一笔订单重复回调只处理一次。
4. **异常兜底**：若前端长期未收到回调，可依赖微信消息推送（订单确认收货事件）由微信推送到后端。

---

## 二、接口约定

### 前端请求

- **POST** `/order/confirm-receive`
- Body：`{ order_number: string, transaction_id?: string }`
- `transaction_id` 有则必传，供后端调微信接口二次校验。

### 后端逻辑概要

1. 幂等：查业务订单是否已「已收货/已完成」，是则直接返回成功。
2. 调微信「查询订单发货状态」：`GET/POST` 微信接口，传 `transaction_id`（或商户号+商户订单号），拿到 `order_state`。
3. 若微信返回 `order_state` 为 3（确认收货）或 4（交易完成），再更新业务订单为已完成；否则返回「微信端未确认收货」类错误。

## 三、Node.js 示例（Express + axios）

```javascript
// 依赖：npm i express axios
const express = require('express')
const axios = require('axios')

const app = express()
app.use(express.json())

// 配置（从环境变量或配置中心读取）
const WECHAT_APPID = process.env.WECHAT_APPID
const WECHAT_SECRET = process.env.WECHAT_SECRET
const WECHAT_MCH_ID = process.env.WECHAT_MCH_ID // 商户号，与发货通知一致

// 获取 access_token（实际项目应缓存，这里简化）
async function getAccessToken() {
  const url = `https://api.weixin.qq.com/cgi-bin/token?grant_type=client_credential&appid=${WECHAT_APPID}&secret=${WECHAT_SECRET}`
  const res = await axios.get(url)
  if (res.data.errcode) throw new Error(res.data.errmsg || 'get token fail')
  return res.data.access_token
}

// 微信「查询订单发货状态」二次校验
// 文档：https://developers.weixin.qq.com/miniprogram/dev/OpenApiDoc/express/express-by-business/getOrder.html
async function getWechatOrderShipping(accessToken, transactionId) {
  const url = `https://api.weixin.qq.com/wxa/sec/order/get_order?access_token=${accessToken}`
  const res = await axios.post(url, { transaction_id: transactionId })
  if (res.data.errcode && res.data.errcode !== 0) {
    throw new Error(res.data.errmsg || '微信查询订单失败')
  }
  return res.data
}

// 从微信返回中解析 order_state（官方在 res.order.order_state，兼容顶层/order_info）
function parseOrderState(wxOrder) {
  if (wxOrder == null) return null
  const raw = wxOrder.order_state ?? wxOrder.order?.order_state ?? wxOrder.order_info?.order_state
  if (raw === undefined || raw === null) return null
  const n = Number(raw)
  return Number.isNaN(n) ? null : n
}

// 业务：根据 order_number 查订单（示例，替换为你的 DB）
async function getOrderByOrderNumber(orderNumber) {
  // return await db.orders.findOne({ order_number: orderNumber })
  return { id: 1, order_number: orderNumber, status: 'pending_recv', transaction_id: '420000xxx' }
}

// 业务：更新订单为已收货（示例，替换为你的 DB）
async function markOrderReceived(orderId, reason) {
  // await db.orders.update({ id: orderId }, { status: 'completed', received_at: new Date() })
  return true
}

// 幂等：检查是否已处理过（示例：用 Redis 或 DB 字段）
async function isOrderAlreadyReceived(orderNumber) {
  const order = await getOrderByOrderNumber(orderNumber)
  return order && ['completed', 'received'].includes(order.status)
}

/**
 * 确认收货接口（前端组件回调后调用）
 * - 幂等：已收货则直接成功
 * - 二次校验：调微信 get_order，仅当 order_state 为 3 或 4 时更新业务
 */
app.post('/order/confirm-receive', async (req, res) => {
  const { order_number, transaction_id } = req.body || {}
  if (!order_number) {
    return res.status(400).json({ code: 400, message: '订单号不能为空' })
  }

  try {
    // 1. 幂等：已收货直接返回
    const alreadyReceived = await isOrderAlreadyReceived(order_number)
    if (alreadyReceived) {
      return res.json({ code: 0, message: '已处理', data: { received: true } })
    }

    // 2. 二次校验：必须用 transaction_id 调微信接口（无则先查业务订单拿 transaction_id）
    let txId = transaction_id
    if (!txId) {
      const order = await getOrderByOrderNumber(order_number)
      txId = order && order.transaction_id
    }
    if (!txId) {
      return res.status(400).json({
        code: 400,
        message: '微信端未确认收货，当前状态：未知。请用户在微信小程序内点击确认收货后再同步。',
      })
    }

    const accessToken = await getAccessToken()

    // 3. 调微信查单，支持延迟重试（微信侧写入可能有延迟）
    let orderState = null
    let wxOrder = null
    const maxRetries = 3
    const retryDelayMs = 1500
    for (let attempt = 1; attempt <= maxRetries; attempt++) {
      wxOrder = await getWechatOrderShipping(accessToken, txId)
      orderState = parseOrderState(wxOrder)
      if (orderState !== null && (orderState === 3 || orderState === 4 || orderState === 6)) break
      if (attempt < maxRetries) {
        console.log(`[confirm-receive] 微信 order_state 未就绪，${retryDelayMs}ms 后重试 (${attempt}/${maxRetries})`, { order_state: orderState, raw: wxOrder })
        await new Promise((r) => setTimeout(r, retryDelayMs))
      }
    }

    // 若仍拿不到有效状态，打日志便于排查（微信返回结构可能与文档不一致）
    if (orderState == null && wxOrder) {
      console.warn('[confirm-receive] 微信 get_order 原始返回（无 order_state）：', JSON.stringify(wxOrder))
    }

    // 4. 仅当微信侧为「确认收货(3)」「交易完成(4)」「资金待结算(6)」时更新业务
    if (orderState === 3 || orderState === 4 || orderState === 6) {
      const order = await getOrderByOrderNumber(order_number)
      if (order) {
        await markOrderReceived(order.id, 'wechat_confirm_receive')
        return res.json({ code: 0, message: '收货已同步', data: { received: true } })
      }
    }

    // 微信侧未到「已收货」
    return res.status(400).json({
      code: 400,
      message: `微信端未确认收货，当前状态：${orderState == null ? '未知状态(None)' : orderState}。请用户在微信小程序内点击确认收货按钮。`,
    })
  } catch (err) {
    console.error('[confirm-receive]', err)
    const msg = err.response && err.response.data && err.response.data.errmsg
      ? err.response.data.errmsg
      : (err.message || '服务器错误')
    return res.status(500).json({ code: 500, message: msg })
  }
})
```

## 四、微信「查询订单发货状态」返回数据结构（get_order）

接口：`POST https://api.weixin.qq.com/wxa/sec/order/get_order?access_token=ACCESS_TOKEN`  
请求体：`{ "transaction_id": "微信支付单号" }` 或 `{ "merchant_id": "商户号", "merchant_trade_no": "商户订单号" }`

**成功时响应（顶层）：**

| 属性       | 类型   | 说明     |
|------------|--------|----------|
| errcode    | number | 错误码，0 为成功 |
| errmsg     | string | 错误原因 |
| **order**  | object | 支付单信息（**order_state 在此对象内**） |

**order 对象字段：**

| 属性              | 类型    | 说明 |
|-------------------|---------|------|
| transaction_id    | string  | 微信订单号 |
| merchant_id       | string  | 商户号 |
| sub_merchant_id   | string  | 二级商户号 |
| merchant_trade_no | string  | 商户订单号 |
| description       | string  | 商品描述（截断 120 字） |
| paid_amount       | number  | 实付金额（分） |
| openid            | string  | 支付者 openid |
| trade_create_time | number  | 交易创建时间戳 |
| pay_time          | number  | 支付时间戳 |
| **order_state**   | **number** | **订单状态：1 待发货 2 已发货 3 确认收货 4 交易完成 5 已退款 6 资金待结算** |
| in_complaint      | boolean | 是否在纠纷中 |
| shipping          | object  | 发货信息（delivery_mode、logistics_type、shipping_list 等） |

**示例响应：**

```json
{
  "errcode": 0,
  "errmsg": "ok",
  "order": {
    "transaction_id": "420000xxx",
    "merchant_id": "1230000109",
    "merchant_trade_no": "ORD20221209132531",
    "description": "商品*1",
    "paid_amount": 100,
    "openid": "ogqztkxxx",
    "trade_create_time": 1670563533,
    "pay_time": 1670563533,
    "order_state": 2,
    "in_complaint": false,
    "shipping": { "delivery_mode": 1, "logistics_type": 1, "shipping_list": [...] }
  }
}
```

**常见解析错误**：微信官方返回里 **没有** 顶层的 `order_state`，只有 `order.order_state`。若后端只读 `res.order_state` 会一直得到 `undefined`/None，从而报「未知状态(None)」。必须从 **order** 对象内取：`res.order.order_state`（示例里已用 `parseOrderState` 兼容顶层、`order`、`order_info`）。

## 五、微信 order_state 参考

| 值 | 含义       |
|----|------------|
| 1  | 待发货     |
| 2  | 已发货     |
| 3  | 确认收货   |
| 4  | 交易完成   |
| 5  | 已退款     |
| 6  | 资金待结算（确认收货后、结算前） |

二次校验通过条件：`order_state === 3 || order_state === 4 || order_state === 6`。

**重要**：微信接口可能把 `order_state` 放在顶层或放在 `order_info` 内，后端需兼容两种结构，例如：`order_state = res.order_state ?? res.order_info?.order_state`。

## 六、「未知状态(None)」与重试

当后端调微信「查询订单发货状态」返回的 `order_state` 为空或异常时，常会报「微信端未确认收货，当前状态：未知状态(None)」。常见原因：

1. **时序**：用户刚在组件里点完「已确认收货」，微信侧尚未写入，后端立刻查单会拿到空/旧状态。
2. **接口/参数**：未传对 `transaction_id`、或用了错误接口/权限，微信返回错误或空数据。

**建议：**

- **后端**：收到前端确认收货请求后，若第一次查微信得到空/非 3、4 状态，可**延迟 1～2 秒重试 1～2 次**再决定是否返回 400；并打日志记录微信原始返回，便于排查。
- **前端**：已在兜底逻辑里「延迟 1.5s 再调后端」、且当报错含「未知状态」时弹窗「状态同步中，请 2～3 秒后重试」并支持点击重试。

---

## 七、幂等与异常兜底

1. **幂等**：用 `order_number`（或 `transaction_id`）在 DB 中判断订单是否已是「已完成」；或使用 Redis 键 `confirm_receive:${order_number}` 设置 24 小时过期，避免重复更新。
2. **异常兜底**：若前端因环境问题长期收不到 `App.onShow` 回调，可在小程序后台配置「消息推送」：选择「订单确认收货」等事件，由微信服务器 POST 到你的后端 URL，后端同样按「查询订单发货状态 → 再更新业务」处理，并做幂等。

## 八、与本项目前端的对接

- 前端已按 2026 流程在 `App.vue` 的 `onShow` 中解析 `referrerInfo.extraData`，并在 `status === 'success'` 时调用 `confirmReceive({ order_number, transaction_id })`。
- 列表/详情页的 onShow 兜底也会传 `transaction_id`（若打开组件时已保存）。
- 后端需实现 `POST /order/confirm-receive`，按上文做二次校验与幂等即可。
